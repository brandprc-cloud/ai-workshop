## ОТВЕТЫ НА КОНКРЕТНЫЕ ВОПРОСЫ

### `?start=lead_250` — что это
**Это deep-link payload Telegram, а НЕ UTM.** Механизм (из `apply.js`):
1. Форма постит JSON на `/api/apply`, backend возвращает `lead_id`.
2. Фронт строит `https://t.me/ArtashesAIBot?start=lead_<lead_id>` и авто-редиректит через 400 мс.
3. При открытии бот получает `/start lead_250` → **привязывает Telegram-пользователя к заявке №250**.
Назначение — **связка «заявка на сайте ↔ пользователь в боте»**, чтобы бот знал, кто пришёл. Это внутренний ключ-джойн, не маркетинговая метка канала.
- Прим.: в боте Lead-id своя нумерация (Андрей = бот-лид 170, payload `lead_250`) — это две разные сущности, связанные payload'ом. Прошлый «сдвиг на 1» — просто две независимые нумерации, не баг.

### `?cohort=2026-06-27` — что это
Параметр выбранной даты, приходит со страницы `/schedule/`. `apply.js` читает его и **преднастраивает выбор даты** в анкете (вставляет сверху radio «Выбрано: 27–28 июня 2026», checked). Даты — из единого `cohorts.json`, прошедшие скрываются автоматически (`cohorts.js`).

### `?ref=XXXXXXXX` — реферальная метка (РАБОТАЕТ машинно!)
- 8 hex-символов. `apply.js` ловит из URL → `sessionStorage` → шлёт `POST /api/ref-click` со стабильным `client_id` (localStorage, дедуп клика на браузер).
- При отправке анкеты `ref` прикладывается к данным. Это **единственная машинная атрибуция канала**, что сейчас есть.

## ГЛАВНЫЙ РАЗРЫВ (vs цель Андрея «каждый клик → CRM, конверсии по шагам»)

1. **GA4 и CRM не связаны.** Сайт-визиты живут в GA4 (анонимно), лиды — в CRM. Сквозную конверсию «визит → анкета → бот → оплата» посчитать НЕЛЬЗЯ: верх воронки (анонимный) не соединён с лидом.
2. **Нет единой идентичности посетителя** через всю цепочку. `client_id` (localStorage) ЕСТЬ, но используется только для дедупа реф-кликов, не как сквозной ключ визит↔лид↔бот.
3. **Нет UTM-атрибуции канала.** Машинно ловится только `ref` (реферал). Источник канала — самоотчёт в `source`, поэтому в CSV 45% пусто/«Другое».
4. **Нет журнала событий воронки** (landing_view, schedule_view, cohort_click, apply_submit, bot_start, payment_chosen, receipt, confirmed, attended) с visitor_id+timestamp+source — а без него нет конверсий по шагам.
5. **Бот показывает прошедшую дату** (13–14 июня) — сайт самочинится, бот нет.

## ЦЕЛЕВОЕ СОСТОЯНИЕ (требования Андрея — записано дословно по смыслу)

Каждое действие должно фиксироваться и давать конверсию по шагу:
- открытие/визит сайта; клик «выбрать дату»/«регистрация»; отправка анкеты; переход в бот; клики в боте; событие платёжки.
- Все источники (сайт, анкета, бот, платёжка) пишут в ОДИН учёт (CRM/таблица событий).
- По каждому шагу видно: (1) количество действий, (2) конверсию в следующий шаг.

**Как это технически закрывается (черновик, для будущего ТЗ):**
- Сквозной `visitor_id` (есть client_id) — ставить на лендинге, протаскивать в `/api/apply` и в `start`-payload бота.
- UTM на ВСЕ ссылки каналов → ловить на лендинге → хранить с visitor_id → тащить в лид.
- Таблица событий воронки (event, visitor_id, lead_id, ts, source) — единый журнал.
- Дашборд: воронка с конверсиями по шагам и по источникам.
